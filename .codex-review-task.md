# HOLP spec 深度 Review — 对照源仓核实版

## 背景

HOLP 是一个新写的 multi-agent 编排协议 spec(v0.1.3,已过四轮自审 review)。这次 review 的重点**不是**协议内部自洽(那个查过了),而是:**spec 声称"借鉴/对位"外部项目的每个地方,是否和那些项目的真实代码一致**。设计者可能凭印象写、可能记错接缝、可能夸大"对位"关系。

**你能读到所有相关仓(都在 /Users/tizer_mac_studio/claude-workspace/ 下,read-only)**:

| 仓 | 路径 | spec 里对应 |
|---|---|---|
| **holp** | `holp/` | 本协议(被 review 对象) |
| **cmux** | `cmux/` | spec 说 events 借鉴 cmux `CmuxEventBus`;flock 替代 cmux `AgentSessionProvider` 硬编码 switch |
| **warp** + **warp-proto-apis** | `warp/`、`warp-proto-apis/` | spec 说 flock/orchestrate 借鉴 Oz proto 的 Harness/RunAgents/ExecutionMode 概念(不抄依赖) |
| **happier** | `happier/` | spec 说朝下 adapter 契约借鉴 happier `ExecutionRunBackendFactory`/`AgentBackend`/`permissionHandler` |
| **loopwright** | `loopwright/` | spec 说 consensus 来自 loopwright 共识评审;治理内核/数据铁三角/状态机待从 loopwright 搬入 |
| **skypilot** / **spawn** | `skypilot/`、`spawn/` | positioning 说 Remote 未来对位 spawn agent×cloud、skypilot 算力调度(V3+) |

## 必读

## Review 任务:逐条核实 spec 的外部声明

对 spec/positioning 里**每一个**对外部项目的声称,去该仓看真代码,判断:**✅ 属实 / 🟡 部分属实(有偏差)/ ❌ 不实(记错/夸大/编造)**。每条给证据(文件:行或代码片段)。

### A. cmux 相关声称

1. spec §5 说 events 订阅"借鉴 cmux `CmuxEventBus`(seq 单调 + replay + 心跳 + slow_consumer)"。
   → 核实 `cmux/Sources/CmuxEventStream.swift` 等:CmuxEventBus 真有 seq/replay/heartbeat/slow_consumer 这套吗?HOLP 的 `events.subscribe`(subscription_id + 带 subscription_id 的 notification)和 cmux 的真实实现是对位、还是 HOLP 借了个名字其实 cmux 不是这么做的?
2. spec §3 说 flock.declare"替代 cmux `AgentSessionProvider` 那种硬编码 switch"。
   → 核实 `cmux/Sources/AgentSessionProvider.swift`:真有 codex/claude/opencode 三家各一个 case 的硬编码吗?HOLP 把它描述成"痛点"准确吗?

### B. Oz / warp 相关声称

3. spec/positioning 说 flock/orchestrate/Local|Remote"借鉴 Oz proto 的概念(不抄依赖、不实现服务端)"。
   → 核实 `warp-proto-apis/apis/multi_agent/v1/orchestration.proto` 和 `task.proto`:真有 `Harness` oneof、`RunAgents`、`ExecutionMode`(Local|Remote)这些 message 吗?HOLP 的对应概念(flock/orchestrate.run/execution_mode)和它们是合理对位、还是 HOLP 改了语义又声称"借鉴"?
4. spec 说 consensus 是 HOLP 独有,"Oz 多 harness 并行但无共识聚合"。
   → 在 warp/warp-proto-apis 里搜:Oz 真的没有跨多 harness 的共识/投票/聚合语义吗?(搜 consensus/quorum/vote/aggregate 相关)别让设计者夸大"独有"。

### C. happier 相关声称

5. adapters/agent-backend.ts 说朝下契约"借鉴 happier `ExecutionRunBackendFactory`/`AgentBackend`/`permissionHandler`"。
   → 核实 `happier/apps/cli/src/agent/executionRuns/registry/`、`happier/apps/cli/src/agent/core/AgentBackend.ts`、AcpBackend 的 `AcpPermissionHandler`:这些接口真实存在、形状和 HOLP 的 `AgentBackend`(startSession/sendPrompt/onMessage/cancel)+ `PermissionHandler`(allow/deny/ask_human)是对位的吗?HOLP 有没有简化掉了 happier 里关键的东西还声称"借鉴"?
6. positioning 说"朝下方言库规划走 happier backends"。
   → 核实 happier 的 backends(`apps/cli/src/backends/`)真能当库调(有干净的 factory 入口)还是必须拉起整个 CLI/daemon?这影响 HOLP"未来接 happier"是否现实。

### D. loopwright 相关声称

7. spec §12 说"从旧 loopwright 搬入:治理内核/数据铁三角/共识/状态机"。
   → 核实 `loopwright/src/`:真有这些东西吗?spec 声称的 `aggregateVerdict`(共识)、events/decisions/registry(数据铁三角)、triage/coder/reviewer 状态机——在 loopwright 里是真有这名字的代码,还是 HOLP 编的?HOLP 的 consensus_verdict 结构能干净映射自 loopwright 的真实聚合逻辑吗?

### E. positioning 的对位关系准确性

8. positioning.md 的"设计来源"表(events←cmux、flock←Oz/happier/spawn、orchestrate←Oz+loopwright、consensus←loopwright、adapter←happier):逐行核实,哪几行属实、哪几行是设计者拔高了对位关系?
9. positioning 说 Remote 未来对位 spawn 的 agent×cloud 矩阵、skypilot 算力调度:核实 spawn 真有 agent×cloud 矩阵(`manifest.json`)、skypilot 真是跨云算力调度,对位描述准确吗?

## 输出格式

1. **逐条核实表**:A/B/C/D/E 的每条,✅/🟡/❌ + 证据(带仓/文件/行)+ 一句话说明。
2. **发现的不实/夸大**(如有):设计者在哪里记错了、夸大了、编造了对位关系。这条最重要——如果 spec 声称"借鉴 X"但 X 其实不是那样,这是诚信问题,要点名。
3. **总判断**:spec 的外部声明整体可信度。有没有需要立刻修正的"虚假对位"。

## 纪律

- 用真代码证据,不要猜。每条判断必须能指到某仓的某文件。
- 不要因为"设计者很努力"就放过不实声称。但属实的就给 ✅,别为挑刺而挑刺。
- 不改任何文件。只输出报告。
