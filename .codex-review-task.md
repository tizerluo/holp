# HOLP 深度 Review 任务书（给 Codex）

## 背景

HOLP (Human On Loop Protocol) 是一个新建的开源、免费、本地优先的 multi-agent 编排协议 + 参考 daemon。定位是 Warp Oz（付费闭源云编排）的**开源对位物**。

本仓刚 init（commit 7b47e0b），目前只有协议层 + 定位 + adapter 契约，**还没有 daemon 实现代码**。你的任务是在写实现之前，深度、批判性地 review 当前这些文档/契约设计，找出真问题——**不要夸奖，要挑毛病**。

## 必读

- `docs/positioning.md` — 定位、市场空洞、设计来源（溯源到 cmux/Oz/happier/spawn/skypilot 五个仓）
- `protocol/spec.md` — v0.1 协议 8 章（核心）
- `protocol/version.md` — 版本策略
- `adapters/agent-backend.ts` — 朝下 agent 适配契约
- `adapters/registry.ts` — v0.1 桩 registry
- `README.md`

## 设计者声称的上下文（你 judge 这些声称是否站得住）

设计者研究了 5 个相关仓后定的协议，声称：
1. events.stream 的形状「照搬 cmux 的 CmuxEventBus（已验证可用）」
2. flock.declare / orchestrate.run「借自 Oz proto 的 Harness oneof / RunAgents」
3. consensus.verdict「是 HOLP 独有、Oz/happier 都没有的协议级能力」
4. 朝下 adapter 契约「借自 happier 的 AgentBackend / permissionHandler 接缝」
5. Local|Remote「进 wire，v0.1 只实现 Local，借 Oz 思路不抄依赖」

## Review 重点（按重要性）

### P0：协议设计的真缺陷
- 协议消息形状有没有内在矛盾、歧义、不可实现之处？
- 版本化/能力降级策略（§9）够不够？consumer 旧版遇到新版 server 会怎样？
- consensus 作为协议参数（§4 + §6）——把"几家同意、排除作者"这种治理语义写进协议层，会不会过耦合？协议该不该知道"作者"这种业务概念？
- Human-on-Loop 的 approval 流（§7）和 events 流（§5）会不会重复/竞争？approval 是 events 的一种，还是独立通道？
- flock.declare（§3）是 push（consumer 主动声明）还是 pull（orchestrator 探测）？cmux 那种"扫进程发现 agent"的 pull 语义协议里没有，是不是漏了？

### P1：与现有生态的契合度
- stdio JSON-RPC 这个传输选型，对 cmux（它现在用 socket-line + 自家 events.stream）、happier（它有完整 RPC 协议）、Warp（它有 proto）来说，是不是真能接？还是设计者乐观了？
- "agent 零改动"这个核心声称——HOLP 朝下要驱动各家，但 v0.1 adapter 全是桩，真实接线（尤其 native-claude 的 stream-json + canCallTool、codex 的 MCP/app-server）有没有被低估？
- 协议名 HOLP / 立场 Human-on-Loop，对外清晰吗？还是又一个难记的缩写？

### P2：实现可行性 / 搬运风险
- 设计者打算从旧 loopwright 仓"挑拷"治理内核/数据铁三角/共识/状态机。这些代码是 TypeScript + drizzle/better-sqlite3。搬进新仓 + 改成"说协议"，有哪些被低估的风险？
- consensus.verdict 的 voters 结构、exclude_author、quorum——旧 loopwright 里的 aggregateVerdict 真能干净映射成协议消息吗？

### 不需要你做的
- 不要改文件。这是 review，不是实现。
- 不要重写 spec。只指出问题 + 给方向。
- 不要夸"设计很好"。要找漏洞。

## 输出格式

按 P0/P1/P2 分级，每条：
- **问题**（一句话）
- **为什么是问题**（证据/推理）
- **方向**（怎么改，但不写代码）

最后给一句总判断：这个 v0.1 协议骨架，**值得照着写实现**，还是**有根本性问题需要先返工**？
