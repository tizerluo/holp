## Review 重点

### A. 返工是否真修对了（逐条核对你之前的 19 条）

设计者声称 v0.1.1 修了(见 spec.md 文末「v0.1 → v0.1.1 变更明细」)。**逐条核实**,每条判断:真修好 / 表面改 / 没改 / 改过头。重点核对:

1. (P0-1) 传输模型:spec §1 现在说「控制面 JSON-RPC + 带 subscription_id 事件 notification」。这个模型内部自洽吗?事件 notification 带 subscription_id 就够了吗?`events.subscribe` 拿 subscription_id、事件带它、`events.unsubscribe` 取消——这套闭环有没有漏洞(比如 subscription 生命周期、server 重启、多 run 路由)?
2. (P0-2) consensus_verdict 现在固定为事件。但 §6 标题还叫「共识裁决」,§2 能力协商里还有没有残留的「不发 consensus.verdict」措辞?全文 consensus 表述一致吗?
3. (P0-3) quorum 可满足性:§6.2 的校验逻辑(`eligible ≥ quorum`)对吗?`eligible` 的定义(排除作者后的可投票家数)清楚吗?有没有边界漏洞(quorum=0、panel 空、全部是作者)?
4. (P0-4) 共识配置合并到 `roles.reviewer` 单一入口。但 §4 示例里 `roles.reviewer` 有 panel+quorum,`policy` 里又有 exclude_author+author_provenance+on_quorum_unsatisfiable——这算不算又拆成两处了?到底共识配置是一处还是两处?
5. (P0-5) 作者降 provenance:`target.produced_by_agent_id` + `policy.author_provenance`。这套够吗?多 artifact 多作者的 run,「排除作者」按哪个 artifact 算?policy 里 author_provenance 有三个取值(produced_by_agent_id/commit_author/run_initiator),但 §6 示例只用了一个,wire 到底认哪个?
6. (P0-6) approval 单通道。§7 说只走事件流。但 §2 能力里 `approval.kinds`、§10 错误码 `approval_required_but_unsupported`、§11 unattended policy 的 approval 超时——这些和「单通道状态机」一致吗?approval_expired/cancelled 状态在 wire 上怎么体现?
7. (P0-7) 能力 feature 细分。§2 的 capabilities 结构够细吗?`consensus`/`approval.{supported,kinds}`/`unattended_loop`/`artifact_refs`/`remote_runner`——有没有该细没细的、或者该是 required 却做成 optional 的?
8. (P0-8) flock.discover。加了,但 declare 和 discover 的关系清楚吗?能不能同时用?discover 探测的真相(版本/登录/凭证)从哪来——daemon 自己跑探测命令?这有没有被低估?

### B. 返工是否引入新问题

- §3 的 flock response(实测能力 status=ready/degraded/rejected)和 §10 错误码(unsupported_transport/missing_auth)——是 flock response 里的 status,还是 orchestrate.run 时的 error?两套机制会不会打架?
- §4.1 Remote 改成 opaque object。但 §4 还说「Remote 留 wire 口子」,§2 capabilities 有 `remote_runner`,§11 没提 Remote。Remote 到底在 v0.1.1 是「定义了但 opaque」还是「完全未定义」?这个边界清楚吗?
- §6.1 的 reviews[] 结构(findings_ref/status/eligible)和 §8 artifact 协议——findings_ref 是 artifact_id 吗?consumer 怎么知道去 artifact.get?引用关系完整吗?
