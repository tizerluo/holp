## Review 重点

### A. 核实「逼到可执行」是否真到位

设计者声称 v0.1.2 把每条「表面改」逼成了可执行状态机/wire schema,或诚实删除/标 v0.2。**逐条核实**,每条标 ✅真可执行 / 🟡还是半截 / ❌没解决 / ⚠️删过头:

1. quorum 两段式(§6.2):「受理时静态上界 = panel 规模 ≥ quorum+1」——这个上界公式对吗?panel 含 rejected/unknown agent 时怎么算?「consensus step 前精确校验」——consensus step 的边界在哪、谁触发这个校验、不满足时除了发 consensus_degraded 事件 run 怎么走(停?跳过?escalate?)?
2. approval 终态(§7):approval_expired/cancelled 事件 + state 字段。终态后 approval.resolve → approval_already_resolved,但 client 怎么知道一个 approval 已经 expired(而不是还在等)?有没有 race(client 在 server 判 expired 的同时 resolve)?cancel 谁能发起(client 还是 server),用什么方法?
3. capability descriptor(§2):{supported, required?, kinds?}。required 的语义清楚吗——client required 一个 server 不支持的,是 initialize 拒绝还是只是 run 级 reject?两方都 required 同一能力但 kinds 不交集呢?descriptor 缺省值(不传某 capability)是什么意思?
4. Remote 删除(§4.1):wire 只剩 Local。那 §11 unattended_policy、§2 capabilities、错误码里还有没有 Remote 残留假设?删 Remote 有没有删掉该留的(比如未来 hook)?
5. artifact 闭环(§8):ref 处必带 envelope。但 §8.2 response 的 content vs content_ref —— 什么条件给 content 什么条件给 content_ref?size 阈值?consumer 怎么知道该期待哪个?content_ref 的 token 怎么用、谁验证?
6. flock 部分成功(§10.1):永远 per-agent status 不抛 error。那 discover 探测失败(比如权限不够跑不了探测命令)算 per-agent rejected 还是整体 error?declare 和 discover 同时用,谁的结果优先?
